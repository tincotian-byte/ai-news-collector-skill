# AI资讯收集器 (AI News Collector)

## 功能概述

自动收集、整理、展示AI行业最新动态，形成结构化的新闻简报系统。

## 工作流程

### 1. 确定存储位置与采集范围

- 优先使用用户选定的工作文件夹，若无则使用 `{workspace}/ai-news/`
- 检查 `news/` 下是否有历史MD文件，读取最近一次采集日期
- **采集范围**：默认采集最近一周的AI新闻。若上次采集未超一周，仅采集上次采集日期之后的新内容，避免重复

### 2. 加载信源配置

读取工作目录下的 `sources.json` 信源配置文件。若不存在，自动从 Skill 目录复制默认版本。
该文件由用户自行维护，可增删信源，格式见"信源配置"章节。

### 3. 搜索AI资讯

**采集优先级：官方一手来源 > 权威科技媒体 > 其他**

**采集方式**：

1. **确定搜索时间范围**
   ```
   采集日期: T_now (如: 2026-03-23)
   搜索起始: T_start = T_now - 7天 (如: 2026-03-16)
   搜索结束: T_end = T_now (如: 2026-03-23)
   ```

2. **搜索时必须限制时间范围**
   - 使用 `after:YYYY-MM-DD before:YYYY-MM-DD` 语法限制搜索时间
   - 示例: `"OpenAI GPT" after:2026-03-16 before:2026-03-23`
   - 绝对禁止无时间限制的搜索

3. **遍历信源采集**
   - 官方来源（tier=1）：WebFetch 访问博客/新闻页，获取最新内容
   - 媒体来源（tier=2）：WebSearch 搜索，必须带时间限制

4. **严格时间过滤（关键）**
   - 即使搜索限制了时间范围，仍需复核每条资讯的事件时间
   - **事件时间必须在搜索时间范围内**（T_start ≤ time ≤ T_end）
   - 若事件时间超出范围 → **丢弃，不收录**
   - 示例：搜索3月16-23日的新闻，发现正文写"1月15日上市" → 丢弃

**关键区分：事件时间 vs 文章时间**：
搜索引擎返回的是"文章发布时间"，但同一事件可能被反复报道。必须区分：
- `time`（事件时间）：事件实际发生的日期，如产品发布日、论文发表日。这是在 HTML 中用于时间筛选的字段
- `date`（采集日期）：Skill 执行采集的日期，用于标识采集批次

**时间复核流程（关键）**：

采集时必须严格复核事件时间，不能直接使用文章发布时间：

1. **官方来源（tier=1）**
   - 官方博客/新闻稿的发布日期通常就是事件时间
   - 直接提取官方声明中的日期

2. **媒体来源（tier=2）——必须复核**
   - **第一步**：WebFetch 获取文章正文全文
   - **第二步**：在正文中搜索时间线索：
     - "XX于3月15日发布..."
     - "本周一（3月10日）..."
     - "昨日/今日" → 结合文章发布时间推算
     - "上周/本月" → 结合上下文判断
   - **第三步**：若正文未明确提及，使用 WebSearch 交叉验证：
     - 搜索 `"XX 发布 日期"`
     - 搜索 `"XX 上市 时间"`
     - 对比多个信源确认

3. **时间复核示例**：

| 文章发布时间 | 正文内容 | 正确的事件时间 | 说明 |
|-------------|---------|--------------|------|
| 2026-03-23 | "智谱AI于1月15日在港交所上市" | 2026-01-15 | 正文明确提及 |
| 2026-03-20 | "百度昨日发布文心4.5" | 2026-03-19 | 推算得出 |
| 2026-03-18 | "腾讯本周一宣布..." | 2026-03-16 | 推算得出 |

4. **绝对禁止**：
   - ❌ 直接将文章发布时间填入 `time` 字段
   - ❌ 不阅读正文就判定时间
   - ❌ 将采集日期作为事件时间

5. **旧事新文过滤**：
   - 若文章报道的是几周/几月前的旧事件 → 不收录
   - 若时间无法确定 → 标记为待核实，暂不收录

### 5. 资讯去重与处理

**去重策略（关键）**：
1. 同一事件多个信源报道 → 仅保留最权威的一条（官方 > 权威媒体 > 其他）
2. 与历史已采集资讯对比 → 读取 `news/` 下最近7天的MD文件，提取已有标题做相似度匹配
3. 标题高度相似（核心事件相同）→ 合并，不重复收录
4. **旧事新文**判定 → 如果一篇新文章报道的是几周/几月前的旧事件（如回顾、盘点），不收录为新资讯

**翻译**：英文标题和摘要翻译为中文，保留原文链接
**分类**：按厂商和技术领域打标签
**优先级**：高（重大发布/突破）、中（常规更新）、低（传闻/次要）

### 6. 生成Markdown简报

文件命名 `YYYY-MM-DD.md`，存入 `news/` 目录。格式见下方模板。

**趋势总结规则（关键）**：

趋势总结必须**严格基于本次采集的新闻内容**进行归纳，禁止引入外部信息或过度推断：

1. **支撑数量要求**：每条趋势必须有**至少2条新闻**作为支撑，单条新闻不能构成趋势
2. **归纳来源**：从新闻的标题、摘要、要点中提取共同特征，进行归类总结
3. **禁止行为**：
   - ❌ 引入未采集的新闻作为趋势依据
   - ❌ 对单条新闻进行过度解读和泛化
   - ❌ 使用"行业普遍认为"、"市场趋势显示"等无法验证的表述
4. **正确示例**：
   - ✅ "多家国产大模型宣布免费开放"（支撑：百度文心、字节豆包等2+条新闻）
   - ✅ "AI Agent能力成为本周发布重点"（支撑：字节豆包Agent、月之暗面Agent等2+条新闻）
   - ❌ "AI推理时代全面到来"（仅NVIDIA一条新闻支撑，过度泛化）
5. **趋势数量**：最多4条，每条用一句话概括，附关键支撑新闻标题

### 7. 生成HTML阅读器和Markdown简报

**架构设计**：生成可直接打开的单文件HTML（数据嵌入），同时保留模板供用户参考。

```
workspace/
├── news/
│   └── 2026-03-23.md          # Markdown格式简报
├── index.html                 # HTML阅读器（数据嵌入，可直接打开）
├── template.html              # HTML模板（参考实现，供用户自定义）
└── sources.json               # 信源配置
```

**生成步骤**：
1. 读取 `news/` 下所有历史MD文件，解析为结构化数据
2. 生成 `news/YYYY-MM-DD.md` Markdown简报
3. **生成 `index.html`** — 将数据和渲染逻辑嵌入单个HTML文件
4. **保留 `template.html`** — 作为参考模板，用户可自行修改样式

**为什么使用数据嵌入方案**：
- 浏览器CORS安全策略禁止 `file://` 协议加载外部JS文件
- 嵌入数据的单文件HTML可以直接双击打开，无需服务器
- 用户如需自定义，可参考 `template.html` 修改后自行生成

**数据格式**：
```javascript
const NEWS_DATA = [
    {
        date: "2026-03-23",       // 采集日期
        title: "资讯标题",
        company: "厂商名",
        field: "技术领域",
        priority: "high/medium/low",
        time: "2026-03-20",       // 事件时间
        source: "来源名称",
        sourceUrl: "来源URL",
        summary: "中文摘要",
        points: ["要点1", "要点2"]
    }
];

const TRENDS_DATA = {
    "2026-03-23": [
        {
            title: "趋势标题",
            summary: "趋势说明",
            sources: ["支撑新闻标题1", "支撑新闻标题2"]
        }
    ]
};

const LAST_UPDATED = "2026-03-23 15:30";
```

**使用方式**：
双击打开 `index.html` 即可浏览所有资讯。支持：
- 顶部时间Tab：今天 / 最近一周(默认) / 最近一个月 / 最近一年 / 自定义
- 趋势区：根据时间范围聚合，最多显示4条
- 侧栏筛选：厂商、技术领域、优先级（分面搜索）
- 视图切换：完整 / 紧凑

**自定义模板**：
如需修改样式，可参考 `template.html` 自行编辑，或告知AI需要的样式调整。

### 8. 最终输出

采集完成后，向用户展示以下信息：

```
✅ AI资讯采集完成

📊 本次采集: X条新资讯
📁 Markdown简报: {workspace}/news/YYYY-MM-DD.md
🌐 HTML阅读器: {workspace}/index.html

📋 可直接访问的链接:
file://{workspace}/index.html
```

**输出要求**：
1. **必须显示完整路径**：`{workspace}/index.html` 的绝对路径
2. **提供file协议链接**：用户可直接点击 `file://...` 链接在浏览器中打开
3. **路径可复制**：确保路径文本便于用户复制粘贴

## 信源配置

采集的信息来源由工作目录下的 `sources.json` 文件管理。Skill首次运行时自动从 `{skill_dir}/sources.json` 复制默认版本到工作目录。

**用户可以随时手动编辑此文件**，增加、删除、启用或禁用信源。

### 信源优先级（tier）

| tier | 含义 | 采集方式 | 可信度 |
|------|------|---------|--------|
| 1 | 官方一手来源 | 优先 WebFetch 直接访问 | 最高，事件时间以此为准 |
| 2 | 权威科技媒体 | WebSearch 关键词搜索 | 高，但需核实事件时间 |
| 3 | 研究机构 | WebSearch 或 WebFetch | 高 |

### sources.json 格式

```json
{
  "sources": [
    {
      "name": "OpenAI",
      "category": "国际AI大厂",
      "tier": 1,
      "enabled": true,
      "urls": ["openai.com/blog"],
      "keywords": ["OpenAI", "GPT", "ChatGPT"],
      "note": "GPT系列、ChatGPT更新"
    }
  ]
}
```

字段说明：
- `name`: 厂商/媒体名称
- `category`: 分类（国际AI大厂 / 国内AI大厂 / 权威科技媒体 / 研究机构 / 用户自定义）
- `tier`: 信源优先级（1=官方, 2=媒体, 3=研究机构）
- `enabled`: 是否启用（false则跳过）
- `urls`: 搜索时用 `site:` 限定的域名
- `keywords`: 搜索时使用的关键词
- `note`: 备注

> **用户增加来源**：直接编辑 `sources.json`，在 `sources` 数组中添加一条即可。也可以告知AI"增加XX为资讯来源"，Skill会自动更新该文件。

## 采集时间逻辑

```
本次采集时间: T_now
上次采集时间: T_last (从 news/ 目录最新文件日期获取)

如果 T_now - T_last < 7天:
    搜索范围 = T_last 到 T_now（仅采集增量）
否则:
    搜索范围 = T_now - 7天 到 T_now（完整一周）

去重范围 = news/ 目录下最近7天的所有MD文件
```


```
├── news/                    # 每日资讯MD文件
│   ├── 2026-03-23.md
```

**文件说明**：
- `news/*.md` — Markdown格式简报，按日期存储
